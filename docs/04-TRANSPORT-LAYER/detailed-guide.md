# 🌐 Транспортный слой (Transport Layer) — Полное руководство

## 🔢 Уровень в модели OSI

| Уровень | Название | Протоколы |
|---------|----------|-----------|
| **7** | Application | HTTP, FTP, SMTP |
| **6** | Presentation | SSL/TLS, JPEG |
| **5** | Session | NetBIOS, RPC |
| **4** | **Transport** | **TCP, UDP, SCTP** |
| **3** | Network | IP, ICMP |
| **2** | Data Link | Ethernet, Wi-Fi |
| **1** | Physical | Кабели, сигналы |

---

## 🎯 Зачем нужен транспортный слой?

Транспортный слой отвечает за:
- ✅ **Надёжность доставки** данных (TCP) или быструю передачу без гарантий (UDP)
- ✅ **Мультиплексирование** — несколько приложений могут использовать один сетевой интерфейс
- ✅ **Контроль потока** — не перегрузить приёмник данными
- ✅ **Обнаружение потерь пакетов** и их повторную отправку

---

## 📊 Основные протоколы

### **TCP (Transmission Control Protocol)** — Надёжная доставка

```
🔹 Гарантирует доставку всех данных
🔹 Устанавливает соединение перед передачей (3-way handshake)
🔹 Контролирует порядок пакетов
🔹 Использует подтверждение ACK и повторную отправку
🔹 Медленнее, но надёжнее
```

**Примеры использования:**
- 🌐 Веб-браузинг (HTTP/HTTPS) — порт 80/443
- 📧 Email (SMTP, POP3, IMAP) — порты 25/110/995
- 💬 Чат (WhatsApp, Telegram) — порт 443
- 📊 FTP/SFTP — порты 21/22

---

### **UDP (User Datagram Protocol)** — Быстрая передача

```
🔹 Не гарантирует доставку
🔹 Нет установки соединения
🔹 Проще и быстрее
🔹 Пакеты могут потеряться или прийти в неправильном порядке
🔹 Идеально для реального времени
```

**Примеры использования:**
- 🎮 Онлайн-игры (позиции игроков) — порт 4334, 7777
- 📹 Видеозвонки (Zoom, Skype) — порты 19302, 8000
- 📡 DNS запросы — порт 53
- 🎵 Стриминг (YouTube Live) — порт 1935

---

### **SCTP (Stream Control Transmission Protocol)** — Современный протокол

```
🔹 Комбинация TCP и UDP
🔹 Поддержка мультигомоинга (несколько IP-адресов в одном соединении)
🔹 Stream-oriented (потоки данных внутри соединения)
🔹 Используется в телефонии (SIP)
```

**Примеры использования:**
- 📞 VoIP (SIP) — порт 5060
- 🌐 Email (SMTP) — порт 25

## 🔄 Как работает TCP?

### **3-way handshake** (установка соединения):

```
┌─────────┐    SYN        ┌─────────┐    SYN+ACK      ┌─────────┐
│  Client │ ───────────▶  │   TCP   │ ─────────────▶  │ Server  │
└─────────┘               └─────────┘                └─────────┘
     A                                                                B

┌─────────┐    ACK        ┌─────────┐    DATA         ┌─────────┐
│ Client  │ ───────────▶  │   TCP   │ ──────────────▶  │ Server  │
└─────────┘               └─────────┘                └─────────┘
     C                                                                D

┌─────────┐    DATA       ┌─────────┐    ACK          ┌─────────┐
│ Client  │ ◀──────────── │   TCP   │ ◀────────────── │ Server  │
└─────────┘               └─────────┘                └─────────┘
     E                                                                F

📊 Результат: Соединение установлено!
```

### **4-way handshake** (закрытие соединения):

```
┌─────────┐    FIN        ┌─────────┐    ACK          ┌─────────┐
│ Client  │ ───────────▶  │   TCP   │ ◀────────────── │ Server  │
└─────────┘               └─────────┘                └─────────┘

┌─────────┐    FIN+ACK    ┌─────────┐    ACK          ┌─────────┐
│ Client  │ ───────────▶  │   TCP   │ ◀────────────── │ Server  │
└─────────┘               └─────────┘                └─────────┘

📊 Результат: Соединение закрыто!
```

## 🎯 Когда использовать TCP?

✅ **Используй TCP, когда:**
- Нужно гарантировать доставку всех данных
- Важна целостность и порядок пакетов
- Передаются важные файлы (документы, базы данных)
- Веб-трафик, email, файловые передачи

❌ **Не используй TCP, если:**
- Нужна минимальная задержка
- Потеря одного пакета не критична
- Данные можно восстановить из других источников

---

## 🎯 Когда использовать UDP?

✅ **Используй UDP, когда:**
- Важнее скорость, чем надёжность
- Потеря пакетов допустима (видео, голос)
- Нужно минимальное время отклика
- Данные восстанавливаются сами (например, в играх)

❌ **Не используй UDP, если:**
- Передаются критически важные данные
- Нужна гарантия доставки
- Порядок данных важен

---

## 📊 Сравнение TCP и UDP

| Характеристика | TCP | UDP |
|----------------|-----|-----|
| **Надёжность** | Гарантирует доставку | Нет гарантий |
| **Соединение** | 3-way handshake | Без соединения |
| **Порядок пакетов** | Да, сортирует | Нет, как пришло |
| **Контроль потока** | Да | Нет |
| **Задержка** | Больше (надежность) | Меньше (быстро) |
| **Размер пакета** | До 65KB | До 64KB |
| **Примеры** | HTTP, FTP, SSH | DNS, VoIP, Gaming |

## 🔧 Практический пример: Веб-сервер и браузер

### **TCP (HTTP):**

```bash
# Браузер отправляет запрос к веб-сайту
curl http://example.com

# Что происходит:
1. Браузер → SYN (хочу соединиться)
2. Сервер → SYN+ACK (согласен)
3. Браузер → ACK (соединение готово)
4. Браузер отправляет HTTP запрос
5. Сервер отвечает HTML страницей
6. Закрытие соединения: FIN, FIN+ACK, ACK, ACK
```

### **UDP (DNS):**

```bash
# Браузер запрашивает IP-адрес домена
nslookup example.com

# Что происходит:
1. Браузер → DNS сервер: "Какой IP у example.com?"
2. DNS сервер → Браузер: "IP: 93.184.216.34"
3. Готово! (без подтверждения)
```

## 🎓 Ключевые термины

| Термин | Определение |
|--------|-------------|
| **Port** | Порт — номер приложения на компьютере (0-65535) |
| **Socket** | Сокет — комбинация IP + порт для соединения |
| **Handshake** | Рукопожатие — процесс установки соединения |
| **ACK** | Acknowledgement — подтверждение получения пакета |
| **SYN** | Synchronize — запрос на установление соединения |
| **FIN** | Finish — запрос на завершение соединения |

## 🔍 Глубокое погружение: Внутренняя работа TCP

### **Сегментация данных:**

TCP разбивает большие данные на сегменты (обычно 1460 байт для Ethernet). Каждый сегмент содержит:
- **Data** — полезная нагрузка
- **Header** — заголовок с метаданными (порты, номер последовательности)

### **Контроль потока:**

TCP использует механизм "скользящего окна" для контроля скорости передачи данных. Если приёмник не успевает обрабатывать данные, он уменьшает размер окна, и отправитель замедляется.

### **Обнаружение потерь пакетов:**

Если TCP не получает подтверждение (ACK) за определённое время, он:
1. Повторно отправляет потерянный сегмент
2. Увеличивает время ожидания (exponential backoff)
3. Если повторные попытки не помогают — соединение разрывается

### **Контроль перегрузки:**

TCP использует алгоритмы для предотвращения перегрузки сети:
- **Slow Start** — быстрое увеличение скорости в начале соединения
- **Congestion Avoidance** — плавное увеличение скорости после достижения порога перегрузки
- **Fast Retransmit** — повторная отправка при потере 3 пакетов подряд

## 🔍 Глубокое погружение: Внутренняя работа UDP

### **Простота:**

UDP не имеет механизма контроля потока или обнаружения потерь. Каждый пакет обрабатывается независимо, что делает его очень быстрым и простым.

### **Преимущества простоты:**
- Меньше накладных расходов (меньший размер заголовка)
- Быстрее обработка на приёмной стороне
- Идеально для приложений реального времени

### **Недостатки:**
- Нет гарантий доставки
- Пакеты могут прийти в неправильном порядке
- Нет механизма повторной отправки потерянных пакетов

## 🛡️ Безопасность транспортного слоя

### **Атаки на TCP:**

1. **SYN Flood** — атака, когда злоумышленник отправляет много SYN-запросов без завершения рукопожатия
   - **Защита:** Rate limiting, SYN cookies

2. **TCP Session Hijacking** — перехват активного соединения
   - **Защита:** Шифрование (TLS), аутентификация

3. **Port Scanning** — сканирование открытых портов для поиска уязвимостей
   - **Защита:** Firewall, IDS/IPS

### **Атаки на UDP:**

1. **UDP Flood** — перегрузка сервера большим количеством UDP-пакетов
   - **Защита:** Rate limiting, фильтрация трафика

2. **DNS Amplification Attack** — использование DNS-серверов для DDoS-атак
   - **Защита:** Rate limiting на DNS-серверах, фильтрация запросов

## 📊 Примеры реального использования

### **Веб-браузинг (HTTP/HTTPS):**

Использует TCP для надёжной передачи HTML, CSS, JavaScript файлов. Каждый ресурс загружается через отдельное TCP-соединение (или несколько соединений для параллелизма).

### **Email (SMTP/POP3/IMAP):**

Использует TCP для гарантированной доставки email сообщений. SMTP использует TCP порт 25 для отправки, POP3/IMAP — порты 110/995 для получения.

### **VoIP (SIP/RTP):**

Использует UDP для передачи голоса в реальном времени. Потеря одного пакета голоса менее критична, чем задержка при ожидании повторной отправки.

### **Онлайн-игры:**

Используют смесь TCP и UDP:
- **TCP** — для синхронизации состояния игры (сохранение прогресса)
- **UDP** — для передачи позиций игроков в реальном времени

## 🎓 Дополнительные протоколы транспортного слоя

### **SCTP (Stream Control Transmission Protocol):**

Комбинация преимуществ TCP и UDP:
- Надёжность доставки как у TCP
- Поддержка мультигомоинга (несколько IP-адресов в одном соединении)
- Stream-oriented (потоки данных внутри соединения)

**Примеры использования:**
- 📞 VoIP (SIP) — порт 5060
- 🌐 Email (SMTP) — порт 25

### **DCCP (Datagram Congestion Control Protocol):**

Компромисс между TCP и UDP:
- Обеспечивает контроль перегрузки как у TCP
- Но не гарантирует доставку как у TCP

**Примеры использования:**
- 📹 Видеостриминг
- 🎵 Аудиопотоки

## 🔧 Практические примеры конфигурации

### **Linux — проверка портов:**

```bash
# Показать все слушающие порты
sudo ss -tlnp

# Проверить конкретный порт:
ss -tlnp | grep :80

# Проверить состояние соединения:
netstat -an | grep ESTABLISHED
```

### **Linux — настройка firewall (iptables):**

```bash
# Разрешить TCP на порту 80 (HTTP)
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Разрешить UDP на порту 53 (DNS)
sudo iptables -A INPUT -p udp --dport 53 -j ACCEPT

# Отклонить все остальные соединения
sudo iptables -P DROP
```

### **Linux — настройка firewall (ufw):**

```bash
# Разрешить HTTP и HTTPS
sudo ufw allow 'OpenSSH'
sudo ufw allow 'Nginx Full'

# Проверить правила:
sudo ufw status verbose
```
