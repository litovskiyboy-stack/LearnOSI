# 🌐 Слой 4: Транспортный уровень (Transport Layer)

## Введение

**Транспортный слой** — это "почта интернета"! Он отвечает за доставку данных от одного приложения к другому на разных компьютерах. Представь, что ты отправляешь письмо другу по почте — транспортный слой делает то же самое для пакетов данных! 📮

---

## 🎯 Основные задачи транспортного уровня:

1. **Надёжная доставка** — гарантировать, что данные дойдут целыми и невредимыми
2. **Разделение потоков** — различать разные приложения (браузер, почта, мессенджер)
3. **Контроль перегрузок** — не засорять сеть слишком большим количеством данных
4. **Мультиплексинг** — несколько приложений могут одновременно отправлять данные

---

## 📦 Ключевые протоколы:

### TCP (Transmission Control Protocol) — надёжный, но медленный
- Гарантирует доставку всех пакетов
- Устанавливает соединение перед передачей данных
- Используется для: веб-сайты (HTTPS), почта, файлы, удалённый доступ

### UDP (User Datagram Protocol) — быстрый, но ненадёжный
- Не гарантирует доставку
- Быстрее TCP (меньше накладных расходов)
- Используется для: видеозвонки, онлайн-игры, стриминг

---

## 🔧 Как работает TCP?

### 3-way handshake (трёхстороннее рукопожатие):

#### Шаг 1: SYN (Synchronize)
```
Клиент → Сервер: "Привет! Я хочу соединиться!"
SYN = 1, Seq = X
```

#### Шаг 2: SYN-ACK (Synchronize-Acknowledge)
```
Сервер → Клиент: "Я тебя слышу! Давай соединимся!"
SYN = 1, ACK = 1, Ack = X+1, Seq = Y
```

#### Шаг 3: ACK (Acknowledge)
```
Клиент → Сервер: "Согласен! Соединение установлено!"
ACK = 1, Ack = Y+1, Seq = X+1
```

**Результат:** Соединение установлено! 🎉

### Почему это важно?
- Гарантирует, что сервер готов принимать данные
- Предотвращает "бомбардировку" неготовых серверов
- Позволяет синхронизировать последовательность пакетов

---

## 📊 Сравнение TCP и UDP:

| Характеристика | TCP | UDP |
|----------------|-----|-----|
| **Надёжность** | Гарантирует доставку | Без гарантий |
| **Скорость** | Медленнее (проверки) | Быстрее |
| **Порядок пакетов** | Сохраняет порядок | Не гарантирует |
| **Установка соединения** | Да (handshake) | Нет |
| **Контроль перегрузок** | Да | Нет |
| **Использование памяти** | Больше | Меньше |

---

## 🎮 Когда использовать TCP?

### Идеальные случаи:
- ✅ Веб-сайты (HTTP/HTTPS) — нельзя потерять данные!
- ✅ Почта (SMTP, IMAP, POP3) — важно каждое письмо
- ✅ Файлы (FTP, SFTP) — нужно передать всё точно
- ✅ Удалённый доступ (SSH) — команды должны прийти в порядке
- ✅ Базы данных — целостность данных критична

### Примеры портов TCP:
| Порт | Протокол | Назначение |
|------|----------|------------|
| 80   | HTTP     | Веб-сайты (незащищённый) |
| 443  | HTTPS    | Веб-сайты (зашифрованный) |
| 21   | FTP      | Передача файлов |
| 22   | SSH      | Удалённый доступ |
| 25, 587 | SMTP   | Отправка почты |

---

## 🎮 Когда использовать UDP?

### Идеальные случаи:
- ✅ Видеозвонки (Zoom, Skype) — лучше потерять кадр, чем ждать!
- ✅ Онлайн-игры — важнее скорость, чем точность
- ✅ Стриминг видео (YouTube Live) — не стоит ждать потерянные пакеты
- ✅ DNS запросы — быстрый ответ важнее надёжности

### Примеры портов UDP:
| Порт | Протокол | Назначение |
|------|----------|------------|
| 53   | DNS      | Перевод доменов в IP |
| 67, 68 | DHCP    | Выдача IP-адресов |
| 123  | NTP      | Синхронизация времени |

## 🔍 Как проверить работу TCP?

### Проверка соединения:

```bash
# Посмотреть открытые порты (TCP)
netstat -tulpn | grep LISTEN

# Или через ss (более быстрый)
ss -tulpn

# Проверить, слушает ли порт
nc -zv google.com 443

# Прослушать порт (как сервер)
nc -l -p 8080
```

### Анализ TCP-соединения:

```bash
# Посмотреть все соединения
netstat -an | grep ESTABLISHED

# Отследить установку соединения
tcpdump -i eth0 port 443 -nn

# Проверить, открыт ли порт с внешнего IP
telnet google.com 80
# или
curl -v https://google.com
```

## 🎯 Практический пример: HTTPS запрос

### Что происходит при посещении сайта?

```
1. Браузер → DNS: "Какой IP у google.com?"
   ↓
2. DNS → Браузер: 142.250.x.x (через UDP порт 53)
   ↓
3. Браузер → Сервер: SYN (порт 80/443 через TCP)
   ↓
4. Сервер → Браузер: SYN-ACK
   ↓
5. Браузер → Сервер: ACK (соединение установлено!)
   ↓
6. Браузер → Сервер: HTTP GET / (через TCP)
   ↓
7. Сервер → Браузер: HTML страница (через TCP)
```

### Почему HTTPS? 🔒

**HTTP (незащищённый):**
```
Браузер: "Привет! Я хочу купить товар за $100"
Сервер: "Спасибо! Ваш номер карты: 1234-5678-9012"
# ВСЕ ДАННЫЕ ВИДНЫ ВООРУЖЁННОМУ ГЛАЗУ! 😱
```

**HTTPS (зашифрованный):**
```
Браузер → Сервер: [ЗАШИФРОВАННЫЙ ПАКЕТ]
Сервер → Браулер: [ЗАШИФРОВАННЫЙ ОТВЕТ]
# Никто не может прочитать данные! 🔒
```

## 🛠️ Настройка TCP параметров (Linux):

### Проверка текущих настроек:

```bash
# Показать все TCP-параметры
sysctl net.ipv4.tcp_*

# Показать количество активных соединений
netstat -an | grep ESTABLISHED | wc -l

# Показать время жизни соединения
cat /proc/sys/net/ipv4/tcp_keepalive_time
```

### Изменение параметров:

```bash
# Увеличить время ожидания закрытого соединения (по умолчанию 60 сек)
sudo sysctl -w net.ipv4.tcp_fin_timeout=30

# Включить пассивное сканирование портов
sudo sysctl -w net.ipv4.tcp_syncookies=1

# Показать текущие настройки
sysctl net.ipv4.tcp_* | head -20
```

## 🎯 Ключевые отличия TCP от UDP:

### Надёжность vs Скорость

| Ситуация | Лучше TCP | Лучше UDP |
|----------|-----------|-----------|
| Отправка важного документа | ✅ Да | ❌ Нет |
| Видеозвонок с другом | ❌ Нет | ✅ Да |
| Онлайн-игра в шутер | ❌ Нет | ✅ Да |
| Загрузка файлов | ✅ Да | ❌ Нет |
| Обновление системы | ✅ Да | ❌ Нет |

## 🎓 Что запомнить:

### TCP — как заказное письмо с уведомлением о вручении:
- Гарантирует доставку
- Можно отследить статус
- Медленнее (проверки)
- Идеально для важных данных

### UDP — как открытка по почте:
- Быстро и просто
- Может потеряться в пути
- Не отслеживается
- Идеально для неважных данных
