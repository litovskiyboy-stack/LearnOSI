## 🧪 Лабораторная работа №1: Физический уровень

### Цель: Проверить физическое соединение

### Теория (простыми словами):
Физический уровень — это как проверить, подключён ли кабель к розетке. Если кабеля нет — интернет не будет работать! 🔌

### Практика:

#### Шаг 1: Проверь состояние кабеля/Wi-Fi

```bash
# Linux/Mac:
ethtool eth0 | grep "Link detected"

# Windows (PowerShell):
Get-NetAdapter | Select-Object Name, LinkSpeed, LinkStatus
```

**Что должно быть:**
- `Link detected: yes` или `LinkStatus: Up`
- Если `no` — значит кабель не подключён! 🔌❌

#### Шаг 2: Посмотри скорость соединения

```bash
# Linux/Mac:
ethtool eth0 | grep Speed

# Windows (PowerShell):
Get-NetAdapter | Select-Object Name, LinkSpeed
```

**Что должно быть:**
- Скорость должна совпадать с тарифом провайдера
- Если скорость низкая — проблема с кабелем или портом! 🐌

#### Шаг 3: Проверь Wi-Fi (если используешь беспроводную связь)

```bash
# Linux/Mac:
iwconfig | grep "ESSID"

# Windows (PowerShell):
Get-NetWiFiConnectionProfile | Select-Object Name, ProfileType
```

**Что должно быть:**
- ESSID — имя твоей Wi-Fi сети
- Если пусто — ты не подключён к Wi-Fi! 📶❌

### ✅ Проверка результата:

Если все команды показывают "Up" или "yes" — физический уровень работает! 🎉

## 🧪 Лабораторная работа №2: Канальный уровень

### Цель: Изучить MAC-адрес и ARP

### Теория (простыми словами):
Канальный уровень отвечает за адресацию внутри одной сети. Каждый компьютер имеет уникальный "паспортный" номер — MAC-адрес! 🎫

### Практика:

#### Шаг 1: Посмотри свой MAC-адрес

```bash
# Linux/Mac:
ip link show eth0 | grep link/ether

# Windows (PowerShell):
Get-NetAdapter -Name "Ethernet" | Select-Object MacAddress
```

**Что должно быть:**
- Формат: `AA:BB:CC:DD:EE:FF` или `00-1A-2B-3C-4D-5E`
- Это твой уникальный номер! 🎫

#### Шаг 2: Посмотри MAC-адрес роутера (шлюза)

```bash
# Linux/Mac:
ip route show | grep default

# Windows (PowerShell):
Get-NetRoute -DestinationPrefix "0.0.0.0/0" | Select-Object NextHop
```

**Что должно быть:**
- IP-адрес твоего роутера (обычно 192.168.1.1 или 192.168.0.1)
- Это "шлюз", через который ты выходишь в интернет! 🌉

#### Шаг 3: Узнай MAC-адрес роутера по его IP

```bash
# Linux/Mac:
arp -a | grep 192.168.1.1

# Windows (PowerShell):
arp -a | Select-String "192.168.1.1"
```

**Что должно быть:**
- Строка с MAC-адресом роутера
- Если пусто — попробуй сначала пинговать роутер: `ping 192.168.1.1`

#### Шаг 4: Посмотри таблицу ARP (кто кому должен данные)

```bash
# Linux/Mac:
arp -a

# Windows (PowerShell):
arp -a
```

**Что должно быть:**
- Список всех устройств в твоей сети с их MAC-адресами
- Это как телефонная книга твоего дома! 📖

### ✅ Проверка результата:

Если ты видишь свои MAC-адреса и адрес роутера — канальный уровень работает! 🎉

## 🧪 Лабораторная работа №3: Сетевой уровень

### Цель: Изучить IP-адреса и маршрутизацию

### Теория (простыми словами):
Сетевой уровень отвечает за адресацию между сетями. Теперь тебе нужен не только номер квартиры, но и улица, дом, город! 🏠

### Практика:

#### Шаг 1: Посмотри свой IP-адрес

```bash
# Linux/Mac:
ip addr show eth0 | grep inet

# Windows (PowerShell):
Get-NetIPAddress -AddressFamily IPv4 | Select-Object IPAddress, InterfaceAlias
```

**Что должно быть:**
- Локальный IP: `192.168.x.x` или `10.x.x.x`
- Это твой "домашний" адрес в сети! 🏠

#### Шаг 2: Посмотри публичный IP (адрес твоего роутера в интернете)

```bash
# Linux/Mac/Windows:
curl ifconfig.me

# Или через Google:
ping -c 1 google.com && grep "from" /proc/net/route
```

**Что должно быть:**
- Публичный IP провайдера (например, `85.234.x.x`)
- Это адрес твоего роутера в интернете! 🌍

#### Шаг 3: Посмотри маршруты (как компьютер находит путь)

```bash
# Linux/Mac:
ip route show

# Windows (PowerShell):
Get-NetRoute -AddressFamily IPv4 | Select-Object DestinationPrefix, NextHop
```

**Что должно быть:**
- Маршрут по умолчанию: `default via 192.168.1.1 dev eth0`
- Это говорит: "Все неизвестные адреса отправляй на роутер!" 🌉

#### Шаг 4: Проверь доступность сайтов (ICMP ping)

```bash
# Linux/Mac/Windows:
ping -c 4 google.com

# Или пинг других сайтов:
ping -c 4 youtube.com
ping -c 4 github.com
```

**Что должно быть:**
- `64 bytes from xxx.xxx.xxx.xxx: icmp_seq=1 ttl=115 time=20 ms`
- Если нет ответа — сайт недоступен или есть блокировка! 🚫

### ✅ Проверка результата:

Если ты видишь свои IP-адреса и сайты отвечают пингом — сетевой уровень работает! 🎉

## 🧪 Лабораторная работа №4: Транспортный уровень

### Цель: Изучить порты и протоколы TCP/UDP

### Теория (простыми словами):
Транспортный уровень отвечает за разбивку больших данных на маленькие пакеты. Представь, что ты отправляешь большой файл — он разбивается на 100 маленьких частей! ✂️

### Практика:

#### Шаг 1: Посмотри открытые порты (слушающие соединения)

```bash
# Linux/Mac:
netstat -tulpn | grep LISTEN

# Windows (PowerShell):
Get-NetTCPConnection -State Listen | Select-Object LocalAddress, LocalPort, OwningProcess
```

**Что должно быть:**
- Порт 22 — SSH (удалённый доступ)
- Порт 80 — HTTP (веб-сайты)
- Порт 443 — HTTPS (защищённые сайты)
- Порт 53 — DNS (перевод адресов в IP)

#### Шаг 2: Проверь соединение с сайтом (порт 443 = HTTPS)

```bash
# Linux/Mac/Windows:
telnet google.com 443

# Или через curl:
curl -v https://google.com | head -10
```

**Что должно быть:**
- `Connected to google.com` или `HTTP/2 200 OK`
- Если ошибка — порт заблокирован или сайт недоступен! 🚫

#### Шаг 3: Посмотри статистику TCP/UDP пакетов

```bash
# Linux/Mac:
cat /proc/net/tcp

# Windows (PowerShell):
Get-NetUDPEndpoint | Select-Object LocalAddress, LocalPort
```

**Что должно быть:**
- Список активных соединений и портов
- Это как журнал доставки писем! 📮

#### Шаг 4: Проверь работу DNS (перевод доменов в IP)

```bash
# Linux/Mac/Windows:
nslookup google.com

# Или через dig:
dig google.com
```

**Что должно быть:**
- `google.com has address 142.250.x.x`
- Это значит, что DNS работает и переводит домен в IP! 🔍

### ✅ Проверка результата:

Если ты видишь открытые порты и сайты отвечают — транспортный уровень работает! 🎉

## 🧪 Лабораторная работа №5: Сессионный уровень

### Цель: Изучить установление соединений

### Теория (простыми словами):
Сессионный уровень отвечает за установление связи перед передачей данных. Как звонок другу: "Привет, давай поговорим?" 📞

### Практика:

#### Шаг 1: Посмотри активные соединения

```bash
# Linux/Mac:
netstat -an | grep ESTABLISHED

# Windows (PowerShell):
Get-NetTCPConnection -State Established | Select-Object LocalAddress, RemoteAddress, State
```

**Что должно быть:**
- Список активных соединений с сайтами
- `ESTABLISHED` — соединение открыто и работает! 🤝

#### Шаг 2: Проверь время установки соединения

```bash
# Linux/Mac/Windows:
curl -v --connect-timeout 5 https://google.com 2>&1 | grep "Connected"
```

**Что должно быть:**
- `Connected to google.com (40.76.x.x) port 443`
- Если долго ждёт — проблема с установкой связи! 🐌

#### Шаг 3: Проверь многоканальность (несколько соединений одновременно)

```bash
# Linux/Mac/Windows:
# Открой несколько вкладок в браузере и посмотри соединения:
netstat -an | grep ESTABLISHED | wc -l
```

**Что должно быть:**
- Несколько активных соединений с разными сайтами
- Это значит, что браузер ведёт параллельные "разговоры"! 🗣️

### ✅ Проверка результата:

Если ты видишь активные соединения и они устанавливаются быстро — сессионный уровень работает! 🎉

## 🧪 Лабораторная работа №6: Представленческий уровень

### Цель: Изучить шифрование и форматы данных

### Теория (простыми словами):
Представленческий уровень отвечает за формат данных. Как переводчик, который переводит текст с русского на английский! Или как шифровальщик, который скрывает содержание письма! 🔒

### Практика:

#### Шаг 1: Проверь шифрование соединения (HTTPS)

```bash
# Linux/Mac/Windows:
curl -v https://google.com 2>&1 | grep "SSL certificate"

# Или через openssl:
openssl s_client -connect google.com:443 -servername google.com
```

**Что должно быть:**
- `SSL certificate verify ok` или `CONNECTED(00000005)`
- Это значит, что соединение защищено! 🔒✅

#### Шаг 2: Проверь сжатие данных (GZIP)

```bash
# Linux/Mac/Windows:
curl -H "Accept-Encoding: gzip" https://google.com | head -10
```

**Что должно быть:**
- Данные должны быть сжаты (меньше размер!)
- Это как упаковать большой чемодан в маленький рюкзак! 🎒

#### Шаг 3: Проверь формат файлов (JPEG, PNG)

```bash
# Linux/Mac/Windows:
file myimage.jpg
file myvideo.mp4
```

**Что должно быть:**
- `myimage.jpg: JPEG image data`
- `myvideo.mp4: QuickTime video data`
- Это значит, что файлы в правильном формате! 📁✅

#### Шаг 4: Проверь работу прокси (если используешь)

```bash
# Linux/Mac/Windows:
curl -v https://google.com | grep "Proxy"
```

**Что должно быть:**
- Если используется прокси — должен быть ответ от прокси-сервера
- Это как отправить письмо через почтовое отделение! 📮

### ✅ Проверка результата:

Если HTTPS работает, данные сжаты и файлы в правильном формате — представленческий уровень работает! 🎉

## 🧪 Лабораторная работа №7: Прикладной уровень

### Цель: Изучить приложения и протоколы

### Теория (простыми словами):
Прикладной уровень — это само приложение, которое ты используешь! Браузер, почтовая программа, мессенджер. Всё, что ты видишь и с чем взаимодействуешь! 🌐

### Практика:

#### Шаг 1: Проверь работу браузера (HTTP/HTTPS)

```bash
# Linux/Mac/Windows:
curl -v https://google.com | head -20
```

**Что должно быть:**
- `HTTP/2 200 OK` или `HTTP/1.1 301 Moved Permanently`
- Это значит, что браузер успешно получил страницу! 🌐✅

#### Шаг 2: Проверь работу почтового клиента (SMTP)

```bash
# Linux/Mac/Windows:
telnet mail.google.com 25

# Или через curl:
curl -v smtp://mail.google.com:25 | head -10
```

**Что должно быть:**
- `220 Gmail SMTP ready` или `250 OK`
- Это значит, что почтовый сервер работает! 📧✅

#### Шаг 3: Проверь работу мессенджера (если есть CLI)

```bash
# Telegram CLI (если установлен):
telegram-cli --help

# WhatsApp Web (через браузер):
# Открой https://web.whatsapp.com и посмотри QR-код
```

**Что должно быть:**
- Мессенджер должен подключиться к серверу
- Это значит, что ты можешь отправлять сообщения! 💬✅

#### Шаг 4: Проверь работу FTP (передача файлов)

```bash
# Linux/Mac/Windows:
ftp ftp.example.com
# Или через curl:
curl -O https://example.com/file.zip
```

**Что должно быть:**
- `Connected to example.com` или файл скачан успешно
- Это значит, что передача файлов работает! 📥✅

### ✅ Проверка результата:

Если браузер открывает сайты, почта отправляет письма и файлы скачиваются — прикладной уровень работает! 🎉
